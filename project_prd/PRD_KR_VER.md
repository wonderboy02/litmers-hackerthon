# PRD_JiraLite_Extended

# AI 기반 이슈 트래킹 웹 애플리케이션 (Jira Lite MVP) - 확장 PRD

---

## 1. 개요

본 과제는 8시간 동안 **AI 기능을 포함한 경량 이슈 트래킹 웹 애플리케이션(MVP)** 을 설계·구현·배포하는 것을 목표로 합니다.

참가자는 요구사항 해석 → 설계 → 구현 → 배포의 전 과정을 수행하며, 이를 통해 **AI-Native 개발 역량, 문제 해결 능력, 웹 서비스 개발 기본기**를 종합적으로 평가합니다.

---

## 2. 핵심 목표

- 불완전한 요구사항을 해석해 **제품화 가능한 구조로 정리하는 능력 평가**
- **이메일 기반 인증, Google OAuth, 팀 관리, 프로젝트/이슈 관리** 등 웹 서비스 기본기 평가
- Drag & Drop UI, AI 요약/제안/자동분류 기능처럼 **AI-Native 프로덕트 개발 능력 평가**
- **대시보드, 알림, 활동 로그** 등 실무 수준의 기능 구현 능력 평가
- 제한된 시간 내에 **배포 가능한 서비스 구현** 능력 확인

---

## 3. 기능 요구사항 상세 (SRS 수준)

이 문서에서는 기능 요구사항을 **FR(Functional Requirement)** 형태로 정의합니다.

---

## 3.1 인증(Authentication)

### FR-001: 회원가입(Sign Up)

사용자는 이메일/비밀번호를 통해 시스템에 회원가입할 수 있어야 한다.

**입력**

| 필드 | 조건 |
| --- | --- |
| `email` | 유니크, 이메일 형식, 최대 255자 |
| `password` | 6자 이상, 최대 100자 |
| `name` | 1~50자 |

**처리**

1. 이메일 중복 체크
2. User 생성
3. 생성 후 자동 로그인 또는 로그인 화면 이동 (자유)

**예외**

- 이메일 중복 → 에러 표시
- 형식 오류 → 에러 표시

---

### FR-002: 로그인(Login)/로그아웃(Logout)

사용자는 이메일/비밀번호로 로그인할 수 있어야 한다.

**입력**

- `email`
- `password`

**처리**

1. 이메일로 사용자 조회
2. 비밀번호 검증
3. 성공 시 세션/토큰 발급 (토큰 만료 시간: **24시간**)
4. 인증 필요한 페이지에 접근 가능

**예외**

- 불일치 → “이메일 또는 비밀번호가 올바르지 않습니다”

**참고**

- 동시 로그인(여러 기기) 허용 여부는 자유롭게 구현

---

### FR-003: 비밀번호 찾기/재설정

사용자는 이메일을 통해 비밀번호를 재설정할 수 있어야 한다.

**처리**

1. 사용자가 이메일 입력
2. 비밀번호 재설정 링크/토큰 생성 (만료 시간: **1시간**)
3. **실제 이메일 발송 필수**
4. 사용자가 새 비밀번호 입력 후 변경 완료

---

### FR-004: Google OAuth 로그인

Google 계정을 사용한 SNS 로그인을 구현해야 한다.

**처리**

1. Google OAuth 인증 흐름 구현
2. 신규 사용자는 자동 회원가입 처리
3. 기존 사용자는 로그인 처리

**주의사항**

- Google OAuth 구현 시, 이메일/비밀번호 기반 로그인과 **별개의 인증 방식**으로 처리
- 동일 이메일로 일반 가입과 Google 로그인 계정 병합은 고려하지 않음

---

### FR-005: 프로필 관리

사용자는 자신의 프로필 정보를 조회하고 수정할 수 있어야 한다.

**수정 가능 항목**

| 필드 | 조건 |
| --- | --- |
| `name` | 1~50자 |
| `profileImage` | 이미지 URL 또는 파일 업로드 |

**처리**

1. 프로필 이미지는 URL 입력 또는 파일 업로드 방식 중 선택 구현
2. 변경 사항 즉시 반영

---

### FR-006: 비밀번호 변경

로그인한 사용자는 비밀번호를 변경할 수 있어야 한다.

**입력**

| 필드 | 조건 |
| --- | --- |
| `currentPassword` | 현재 비밀번호 |
| `newPassword` | 6자 이상, 최대 100자 |
| `confirmPassword` | newPassword와 일치 |

**처리**

1. 현재 비밀번호 검증
2. 새 비밀번호로 변경
3. 변경 완료 메시지 표시

**예외**

- 현재 비밀번호 불일치 → 에러 표시
- 새 비밀번호 확인 불일치 → 에러 표시

**참고**

- Google OAuth로만 가입한 사용자는 이 기능 비활성화

---

### FR-007: 계정 삭제

사용자는 자신의 계정을 삭제할 수 있어야 한다.

**처리**

1. 비밀번호 재확인 (OAuth 사용자는 확인 버튼만)
2. 소유한 팀 존재 여부 확인
3. 소유한 팀이 있으면 삭제 불가 → 팀 삭제 또는 소유권 이전 안내
4. 계정 및 관련 데이터 Soft Delete 처리

**예외**

- 소유한 팀 존재 시 → “소유한 팀을 먼저 삭제하거나 소유권을 이전해주세요”

---

## 3.2 팀(Team) 기능

### FR-010: 팀 생성(Create Team)

로그인한 사용자는 새로운 팀을 생성할 수 있어야 한다.

**입력**

| 필드 | 조건 |
| --- | --- |
| `name` | 1~50자 |

**처리**

1. Team 생성 (`ownerId = 현재 사용자`)
2. TeamMember 생성 (role = OWNER)

**참고**

- 한 사용자는 **여러 팀에 소속 가능**
- 팀 역할은 **OWNER / ADMIN / MEMBER** 세 가지 존재

---

### FR-011: 팀 정보 수정(Update Team)

**권한**: 팀 OWNER, ADMIN

**수정 가능 항목**

- 팀 이름

---

### FR-012: 팀 삭제(Delete Team)

**권한**: 팀 OWNER만 가능

**처리**

- 팀 삭제 시 하위 프로젝트, 이슈, 댓글 등 **모두 Soft Delete 처리**
- 삭제된 팀은 30일간 복구 가능 (복구 기능은 선택)

**참고**

- OWNER는 팀 탈퇴 불가, 삭제만 가능

---

### FR-013: 팀 멤버 초대(Invite Member)

**권한**: 팀 OWNER, ADMIN

**처리 방식** (둘 중 자유롭게 선택)

1. 초대 이메일 입력 → 초대 정보 저장 → 해당 이메일로 가입 시 자동 팀 합류
2. 초대 이메일 입력 → 초대받은 사용자가 “초대 목록” 화면에서 승인 → 팀 합류

**초대 규칙**

| 항목 | 내용 |
| --- | --- |
| 초대 만료 기간 | **7일** |
| 초대 거절 기능 | 없음 (미수락 시 pending 상태 유지) |
| 재발송 | 가능 (기존 pending 초대의 만료일 갱신) |

**이메일 발송**

- **실제 이메일 발송 필수**

---

### FR-014: 팀 멤버 조회(View Members)

팀에 속한 사용자 목록을 조회할 수 있어야 한다.

**표시 정보**

- 멤버 이름, 이메일, 역할(OWNER/ADMIN/MEMBER), 가입일

---

### FR-015: 팀 멤버 강제 퇴장(Kick Member)

**권한**:
- OWNER: 모든 멤버 강제 퇴장 가능
- ADMIN: MEMBER만 강제 퇴장 가능

**처리**

- 해당 멤버를 팀에서 제거
- OWNER/ADMIN 본인은 강제 퇴장 불가

---

### FR-016: 팀 탈퇴(Leave Team)

**권한**: ADMIN, MEMBER만 가능 (OWNER는 탈퇴 불가, 삭제만 가능)

**처리**

- 해당 사용자를 팀에서 제거

---

### FR-017: 팀 역할 체계 (OWNER/ADMIN/MEMBER)

팀 내 역할은 3단계로 구분된다.

**역할 정의**

| 역할 | 설명 |
| --- | --- |
| OWNER | 팀 생성자, 최고 권한, 팀당 1명 |
| ADMIN | 관리자 권한, 멤버 관리 가능 |
| MEMBER | 일반 멤버, 프로젝트/이슈 작업 가능 |

---

### FR-018: 역할 변경(Change Role)

**권한**: OWNER만 가능

**처리**

1. MEMBER → ADMIN 승격
2. ADMIN → MEMBER 강등
3. OWNER 권한 이전 (다른 ADMIN에게 OWNER 양도 시 본인은 ADMIN으로 변경)

**제한**

- OWNER는 최소 1명 유지 필수

---

### FR-019: 팀 활동 로그(Team Activity Log)

팀 내 주요 활동을 시간순으로 조회할 수 있어야 한다.

**기록 대상**

- 멤버 가입/탈퇴/강제 퇴장
- 역할 변경
- 프로젝트 생성/삭제/아카이브
- 팀 정보 수정

**표시 정보**

- 활동 내용, 수행자, 대상, 일시

**페이지네이션**

- 무한 스크롤 또는 페이지네이션 구현 필수

---

## 3.3 프로젝트(Project)

### FR-020: 프로젝트 생성(Create Project)

사용자는 자신이 소속된 팀 내에 프로젝트를 생성할 수 있어야 한다.

**입력**

| 필드 | 필수 | 조건 |
| --- | --- | --- |
| `name` | 필수 | 1~100자 |
| `description` | 선택 | 최대 2000자 |

**처리**

1. Project 생성 (`teamId` = 현재 팀, `ownerId` = 생성자)
2. 생성된 프로젝트는 팀 멤버 전체가 열람 가능

**제한**

- **팀당 최대 15개** 프로젝트

**참고**

- 프로젝트 소유자 = 생성한 사람

---

### FR-021: 프로젝트 목록 조회(View Projects)

사용자는 자신이 속한 팀의 프로젝트 전체를 조회할 수 있어야 한다.

**표시 정보**

- 프로젝트 이름, 설명 (요약), 이슈 개수, 생성일, 즐겨찾기 여부

**정렬**

- 즐겨찾기 프로젝트 우선 표시
- 이후 생성일 역순

---

### FR-022: 프로젝트 상세 페이지(Project Detail)

프로젝트 정보 + 해당 프로젝트의 이슈 목록/칸반 보드 표시.

**표시 정보**

- 프로젝트 이름, 설명
- 이슈 통계 (상태별 개수)
- 칸반 보드 또는 이슈 리스트 (탭 전환)

---

### FR-023: 프로젝트 수정(Update Project)

**권한**: 팀 OWNER, ADMIN, 또는 프로젝트 소유자

**수정 가능 항목**

- 프로젝트 이름
- 프로젝트 설명

---

### FR-024: 프로젝트 삭제(Delete Project)

**권한**: 팀 OWNER, ADMIN, 또는 프로젝트 소유자

**처리**

- 프로젝트 삭제 시 하위 이슈, 댓글 등 **모두 Soft Delete 처리**

---

### FR-025: 프로젝트 설명(Project Description)

프로젝트는 설명 필드를 가진다.

**조건**

- 최대 2000자
- 마크다운 지원 권장

---

### FR-026: 프로젝트 아카이브(Archive Project)

**권한**: 팀 OWNER, ADMIN, 또는 프로젝트 소유자

**처리**

1. 프로젝트를 아카이브 상태로 변경
2. 아카이브된 프로젝트는 목록에서 별도 표시 또는 숨김
3. 아카이브된 프로젝트의 이슈는 읽기 전용

**복원**

- 아카이브된 프로젝트는 언제든 복원 가능

---

### FR-027: 프로젝트 즐겨찾기(Favorite Project)

사용자는 프로젝트를 즐겨찾기할 수 있어야 한다.

**처리**

1. 즐겨찾기 토글 (추가/해제)
2. 즐겨찾기한 프로젝트는 목록 상단에 표시

**참고**

- 즐겨찾기는 사용자별로 관리됨

---

## 3.4 이슈(Issue)

### FR-030: 이슈 생성(Create Issue)

프로젝트 안에서 이슈를 생성할 수 있어야 한다.

**입력**

| 필드 | 필수 | 조건 |
| --- | --- | --- |
| `title` | 필수 | 1~200자 |
| `description` | 선택 | 최대 5000자 |
| `assigneeUserId` | 선택 | 같은 팀 멤버여야 함 |
| `dueDate` | 선택 | 날짜 형식 |
| `priority` | 선택 | HIGH/MEDIUM/LOW (기본: MEDIUM) |
| `labels` | 선택 | 프로젝트 라벨 중 복수 선택 |

**처리**

- 생성 시 `status = Backlog`
- 이슈 소유자 = 생성한 사람

**제한**

- **프로젝트당 최대 200개** 이슈

---

### FR-031: 이슈 상세 조회(Issue Detail)

다음 정보가 표시되어야 한다:

- 제목
- 설명
- 상태
- 우선순위
- 담당자
- 마감일
- 라벨
- 생성일
- 서브태스크 목록
- 댓글 리스트
- AI 요약 (버튼)
- AI 제안 (버튼)
- 변경 히스토리 (버튼 또는 탭)

---

### FR-032: 이슈 수정(Update Issue)

**권한**: 팀 멤버 전체

**수정 가능 항목**

- 제목, 설명, 담당자, 마감일, 상태, 우선순위, 라벨

---

### FR-033: 이슈 상태 변경(Update Status)

Drag & Drop 또는 상세 화면에서 상태 변경 가능.

**기본 상태**

- `Backlog`
- `In Progress`
- `Done`

**커스텀 상태**

- 프로젝트별로 추가 상태 정의 가능 (FR-053 참조)

**상태 전이 규칙**

- 모든 상태 간 **직접 이동 가능** (순차적 제한 없음)

잘못된 상태값으로 변경 시도 시 오류 처리.

---

### FR-034: 담당자 지정(Assign User)

`assigneeUserId`는 해당 프로젝트의 **팀 멤버 중 한 명**이어야 한다.

---

### FR-035: 이슈 삭제(Delete Issue)

**권한**: 이슈 소유자(생성자), 프로젝트 소유자, 팀 OWNER, 팀 ADMIN

**처리**

- Soft Delete 처리

---

### FR-036: 이슈 검색/필터링

**검색**

- 제목 텍스트 검색

**필터**

- 상태별 (Backlog / In Progress / Done / 커스텀)
- 담당자별
- 우선순위별 (HIGH / MEDIUM / LOW)
- 라벨별
- 마감일 유무
- 마감일 범위

**정렬**

- 생성일, 마감일, 우선순위, 최근 수정일

---

### FR-037: 이슈 우선순위(Priority)

이슈는 우선순위를 가진다.

**우선순위 레벨**

| 레벨 | 설명 |
| --- | --- |
| HIGH | 긴급, 즉시 처리 필요 |
| MEDIUM | 일반적인 우선순위 (기본값) |
| LOW | 낮은 우선순위 |

**표시**

- 칸반 보드 및 이슈 목록에서 시각적 구분 (색상, 아이콘 등)

---

### FR-038: 이슈 라벨/태그(Labels)

프로젝트별로 커스텀 라벨을 생성하고 이슈에 적용할 수 있어야 한다.

**라벨 관리**

| 필드 | 조건 |
| --- | --- |
| `name` | 1~30자 |
| `color` | HEX 색상 코드 |

**처리**

1. 프로젝트 내에서 라벨 생성/수정/삭제
2. 이슈에 라벨 복수 적용 가능

**제한**

- **프로젝트당 최대 20개** 라벨
- **이슈당 최대 5개** 라벨 적용

---

### FR-039: 이슈 변경 히스토리(Issue History)

이슈의 변경 이력을 조회할 수 있어야 한다.

**기록 대상**

- 상태 변경
- 담당자 변경
- 우선순위 변경
- 제목 변경
- 마감일 변경

**표시 정보**

- 변경 항목, 이전 값, 새 값, 변경자, 변경 일시

### FR-039-2: 서브태스크(Sub-task)

**입력**

**처리**

2. 체크박스로 완료/미완료 표시
3. 드래그로 순서 변경 가능

**제한**

- **이슈당 최대 20개** 서브태스크

**표시**

## 3.5 AI 기능

### FR-040: 설명 요약 생성(AI Summary)

이슈의 description을 LLM API에 전달해 2~4문장 요약을 생성해야 한다.

**동작 방식**

- **버튼 클릭 시** 생성 (자동 생성 아님)

**캐싱**

- 한 번 생성된 AI 결과는 **저장(캐싱)**
- description이 수정되면 **캐시 무효화** (재생성 필요)

**제한**

- description이 **10자 이하**이면 AI 기능 실행 불가 (버튼 비활성화 또는 안내 메시지)

**에러 처리**

- AI API 호출 실패 시 에러 메시지 표시

---

### FR-041: 해결 전략 제안(AI Suggestion)

“이 이슈를 해결하기 위한 접근 방식을 제안해줘”와 같은 요청을 LLM에 보내고 결과를 표시해야 한다.

**동작 방식**

- **버튼 클릭 시** 생성

**캐싱**

- FR-040과 동일하게 캐싱 및 무효화 처리

**제한**

- description이 **10자 이하**이면 실행 불가

**에러 처리**

- AI API 호출 실패 시 에러 메시지 표시

---

### FR-042: AI Rate Limiting

AI API 호출에 대한 rate limiting을 구현해야 한다.

**정책**

- 사용자당 분당 10회 제한
- 또는 일당 100회 제한
- 둘 중 하나 이상 구현 필수

**처리**

- 제한 초과 시 에러 메시지 및 남은 시간/횟수 안내

---

### FR-043: AI 이슈 자동 분류(AI Auto-Label)

이슈 생성 시 제목과 설명을 기반으로 적절한 라벨을 자동 추천해야 한다.

**동작 방식**

1. 이슈 생성 시 “AI 라벨 추천” 버튼 표시
2. 클릭 시 LLM이 프로젝트 내 기존 라벨 중 적합한 것 추천
3. 사용자가 추천 라벨 수락/거부 선택

**입력**

- 이슈 제목, 설명
- 프로젝트 내 라벨 목록

**출력**

- 추천 라벨 (최대 3개)

---

### FR-044: AI 중복 이슈 탐지(AI Duplicate Detection)

새 이슈 생성 시 기존 이슈와 유사한 이슈를 탐지하여 경고해야 한다.

**동작 방식**

1. 이슈 생성 폼에서 제목 입력 완료 시 (또는 버튼 클릭)
2. 기존 이슈들과 유사도 비교
3. 유사한 이슈가 있으면 경고 표시

**출력**

- 유사 이슈 목록 (최대 3개)
- 각 이슈로 이동 가능한 링크

**처리**

- 사용자가 경고를 무시하고 생성 가능

---

### FR-045: AI 댓글 요약(AI Comment Summary)

댓글이 많은 이슈에서 논의 내용을 요약해야 한다.

**동작 방식**

- **버튼 클릭 시** 생성
- 댓글이 **5개 이상**일 때만 버튼 활성화

**캐싱**

- 새 댓글 추가 시 **캐시 무효화**

**출력**

- 논의 요약 (3~5문장)
- 주요 결정 사항 (있는 경우)

---

**이슈 카드 표시 정보**

- 담당자

- 생성일

---

### FR-052: 같은 컬럼 내 순서 변경

같은 상태(컬럼) 내에서 이슈 순서를 위아래로 드래그하여 변경할 수 있어야 한다.

**처리**

- 새 이슈는 컬럼 최하단에 추가

---

### FR-053: 커스텀 컬럼(Custom Status)

프로젝트별로 기본 상태 외에 추가 상태를 정의할 수 있어야 한다.

**입력**

| 필드 | 조건 |
| --- | --- |
| `name` | 1~30자 |
| `color` | HEX 색상 코드 (선택) |
| `position` | 컬럼 순서 |

**제한**

- 기본 3개 + **최대 5개** 커스텀 상태 (총 8개)

**처리**

1. 커스텀 상태 생성/수정/삭제
2. 상태 삭제 시 해당 상태의 이슈는 Backlog로 이동

---

### FR-054: WIP Limit (Work In Progress Limit)

컬럼별로 최대 이슈 개수를 제한할 수 있어야 한다.

**입력**

| 필드 | 조건 |
| --- | --- |
| `wipLimit` | 1~50 또는 무제한(null) |

**처리**

1. WIP Limit 설정/해제
2. 제한 초과 시 경고 표시 (이동은 허용하되 시각적 경고)

**표시**

- 컬럼 헤더에 현재 이슈 수 / WIP Limit 표시 (예: 5/10)
- 초과 시 컬럼 헤더 강조 (색상 변경 등)
